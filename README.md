# brick-bootkit-admin

`brick-bootkit-admin` 是一套面向企业中后台场景的可扩展平台，目标不是只提供一个“能跑起来”的后台模板，而是帮助团队把后台系统从“快速上线”推进到“可持续演进、可持续治理、可持续交付”。

一句话理解它：

> 用一套平台同时解决后台建设、扩展接入、安装交付和运维治理问题。

它强调三件事：

- 后台能力要能快速落地
- 扩展能力要能持续接入
- 上线与运维过程要能被治理

---

## 项目简介

在很多企业项目里，后台系统一开始只是为了解决“先把功能做出来”，但随着业务增长，问题会很快暴露：

- 角色和权限越来越复杂
- 新业务模块越来越多
- 前后端能力需要拆分协作
- 插件、微应用、智能服务需要不断接入
- 安装、部署、升级、回滚越来越容易出错

`brick-bootkit-admin` 面向的就是这样的阶段性问题。  
它把后台基础能力、扩展能力、安装交付能力和治理能力放到同一套平台路径里，让团队不必在“快速上线”和“长期可维护”之间二选一。

---

## 为什么选择它

如果你要解决的是“做一个长期可维护、可持续交付、可持续扩展的后台体系”，而不是只搭一个演示型后台，那么这个项目的定位会更贴近真实业务需求。

和常见后台模板相比，它更强调以下能力组合：

| 维度 | 常见后台模板 | brick-bootkit-admin |
| --- | --- | --- |
| 快速起步 | 通常具备 | 具备 |
| 权限与组织治理 | 基础支持 | 面向复杂后台持续治理 |
| 插件/微应用/MCP 接入 | 往往需要自行拼装 | 作为平台能力统一纳入 |
| 安装与交付 | 依赖人工说明 | 支持安装向导、脚本、打包、Release |
| 多环境部署 | 常需自行适配 | 支持 Docker / Native / 多依赖模式 |
| 安装过程追踪 | 通常缺失 | 支持预检、执行、追踪、重试 |
| 持续演进 | 后期容易失控 | 从一开始就考虑扩展与治理并行 |

如果你的目标是：

- 让后台能持续接业务
- 让扩展能力能持续接入
- 让交付和运维不再成为上线瓶颈

那么它的价值会比纯模板型后台更明显。

## 解决什么痛点

### 1. 后台系统起步快，但后续越来越难维护

很多后台模板前期看起来够用，但一旦业务复杂度上来，就会出现：

- 权限模型难扩展
- 菜单、组织、角色边界混乱
- 模块越加越乱，维护成本越来越高

`brick-bootkit-admin` 的目标不是只让你“起步快”，而是让后台体系在后续增长阶段仍然保持可管理、可演进。

### 2. 扩展能力接入快，但治理能力跟不上

当插件、微应用、MCP 服务、应用包不断接入后，如果缺少统一入口和任务治理，就会出现：

- 新能力能接进来，但不能稳定运维
- 不同扩展缺少统一状态管理
- 安装失败、升级失败时难追踪、难重试

这个项目把“扩展能力”和“治理能力”放在同一平台中考虑，而不是把扩展接入当成一次性动作。

### 3. 安装部署、环境准备、版本发布链路割裂

企业后台的痛点很多时候不在代码本身，而在交付过程：

- 依赖环境复杂，MySQL / Redis / Docker / 静态资源常常不一致
- 客户环境差异大，发布方式很难统一
- 首装、升级、回滚、健康验证缺少闭环

`brick-bootkit-admin` 强调安装预检、环境准备、安装任务追踪和多模式部署，尽量把交付过程标准化。

### 4. 系统能上线，但安全与风险控制总是后补

很多系统是先把功能推上去，再逐步补：

- 请求安全
- 权限控制
- 安装风险阻断
- 运行状态追踪
- 问题定位与审计

这个项目更强调把这些治理能力前置到平台层，而不是等上线后再补丁式修复。

## 核心能力

## 1. 基础后台能力

用于支撑企业日常后台管理场景：

- 角色、菜单、权限治理
- 组织与部门结构管理
- 统一后台入口与管理视图
- 运营后台常见模块的承载能力

这部分能力解决的是“后台基础盘”的问题，让系统先具备稳定的管理底座。

## 2. 可扩展生态能力

项目不是封闭后台，而是面向持续扩展设计：

- 插件能力接入
- 微应用能力接入
- MCP 服务接入
- 应用包、版本与安装任务管理

这意味着业务可以按阶段逐步扩展，而不是每次都重做一套后台。

## 3. 安装与交付能力

这是项目非常重要的一部分：

- 支持安装向导
- 支持环境预检
- 支持内置 / 外部 MySQL、Redis
- 支持 Docker 与原生运行模式
- 支持多平台打包与 Release 发布
- 支持交付 payload、安装脚本、服务注册模板统一组织

它解决的是“项目能做出来，但交付总是出问题”的现实痛点。

## 4. 安全与治理能力

平台能力不只关注功能，也关注上线与运维风险：

- 请求/响应加密协议能力
- 授权开关与能力控制
- 安装前预检
- 安装任务状态跟踪
- 失败重试与问题定位
- 健康检查、运行追踪、日志治理

这部分能力的价值在于：让系统不仅能上线，还能更稳地上线。

## 5. 持续发布与版本管理能力

围绕交付闭环，项目还强调：

- 多平台制品输出
- 统一 payload 结构
- Release 资产组织
- 升级、回滚、校验与版本追踪

对于需要持续交付的团队，这部分能力会直接影响后续成本。

## Release 与交付

本项目适合作为 Release 发布入口，建议发布以下资产：

- 后端可执行 Jar
- payload 交付包
- Windows 安装包（`.exe`）
- macOS 安装包（`.dmg`）
- Linux 包（`.deb` / `.rpm` / `.tar.gz`）
- 校验文件与版本说明

推荐把 Release 做成“可以直接交付”的形式，而不是只上传源码压缩包。  
如果你的目标用户是实施人员、运维人员或客户现场团队，这一点尤其重要。

建议同时区分以下附件：

- `jar`
  - 适合已有运行环境、希望自行部署的用户
- `payload`
  - 适合实施、运维、二次封装和首装场景
- `windows-exe`
  - 适合 Windows 现场交付
- `macos-dmg`
  - 适合 macOS 本地安装
- `linux-deb/rpm/tar.gz`
  - 适合 Linux 运维部署

推荐的首装可用交付结构：

```text
payload/
  app/
    brick-bootkit-admin.jar
  webapps/
    webroot/
    childrens/   # optional
  config/
    application-product.yml
  scripts/
  installer/
  configs/
    services/
```

这套结构适合：

- 直接运行
- 首装引导
- 环境准备
- 服务注册
- 升级与运维

## 授权与套餐说明

`brick-bootkit-admin` 不是一个完全免费的系统。

这样设计并不是为了制造复杂门槛，而是因为我们希望：

- 真正有使用需求的人进入系统
- 持续维护、持续迭代、持续交付能够有基本保障
- 避免“零成本拿走、零反馈使用、零责任消耗资源”的白嫖式使用

这个系统从最初规划到逐步成型，前后经历了约 **3 年** 的时间。  
它不是一次性拼出来的演示项目，而是在多轮真实业务验证、功能打磨、交付实践和问题复盘中逐步沉淀出来的结果。

也正因为如此，我们不希望它最后变成“随便拿走试用、没有任何投入、没有任何沉淀反馈”的项目。  
为了给真正的使用者设置合理门槛，同时也保障后续版本维护与交付支持，我们采用 **按时长收费** 的授权方式。

### 为什么采用按时长收费

采用按时长收费，而不是一次性统一高价，主要是为了让门槛“可进入、可选择、可持续”：

- 对初次评估的团队来说，月度和季度成本足够低，可以先验证是否适合自己的业务
- 对已经稳定使用的团队来说，半年版、年度版和长期版更适合长期投入
- 对项目本身来说，这种方式也更有利于持续维护、版本发布、问题修复和交付支持

### 套餐选择建议

如果你还在评估阶段，可以按下面的方式理解这些套餐：

- **月度体验版**
  - 适合个人试用、功能体验、短周期验证
- **季度版**
  - 适合小团队试运行、短项目交付、阶段性内部落地
- **半年版**
  - 适合已经准备进入稳定使用期，但还没有长期锁定的团队
- **年度版**
  - 适合已经明确要长期使用，并希望按自然年度管理授权周期的团队
- **长期版**
  - 适合确定会长期部署、长期维护、长期作为业务底座使用的团队

### 购买后怎么使用

项目当前采用的是“按时长授权”的使用方式。  
通常可以按下面的流程理解：

1. 选择适合你的授权时长
2. 获取对应的授权码或授权凭据
3. 在系统安装或授权入口导入授权信息

它的目的不是增加人为障碍，而是确保“真正有使用需求的人”进入系统，同时保证后续维护、更新和交付支持是可持续的。

### 适合哪些购买路径

你可以把授权决策理解成这样：

  - 直接半年版或年度版会更合适
- **已经明确会长期部署**
  - 长期版更适合长期运营、持续迭代和稳定交付场景

### 适合什么样的团队

如果你属于以下类型，按时长授权通常更合适：

- 想先低成本验证是否适合自己业务的团队

- 需要持续迭代、长期使用而不是一次性试用的团队
- 重视安装交付、升级维护和版本持续可用性的团队

### 我们的态度

我们更希望这个系统服务于：

- 真正准备部署和使用的人
- 愿意把后台体系做成长期能力的人

- 快速搭建后台
- 后续持续扩展
- 把安装、交付、升级、治理一起纳入体系

那么 `brick-bootkit-admin` 会比单纯的后台模板更适合长期投入。
