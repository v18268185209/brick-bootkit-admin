## 附件说明

- `brick-bootkit-admin.jar`
  - 适合已有 Java 与运行环境、希望自行部署的用户

- `payload`
  - 适合实施、运维或二次封装场景
  - 通常包含：
    - `app/`
    - `webapps/`
    - `config/`
    - `scripts/`
    - `installer/`
    - `configs/services/`

- `windows-*.exe`
  - 适合 Windows 环境直接安装

- `macos-*.dmg`
  - 适合 macOS 本地交付

- `linux-*.deb / *.rpm / *.tar.gz`
  - 适合 Linux 运维环境

## 下载哪个包最合适

如果你不确定该下载哪个附件，可以按下面理解：

- 你已经有自己的运行环境，只想自行部署：
  - 选 `brick-bootkit-admin.jar`

- 你希望拿到一套完整交付目录，便于实施或客户现场部署：
  - 选 `payload`

- 你在 Windows 环境，希望尽量简化安装步骤：
  - 选 `windows-*.exe`

- 你在 macOS 环境本地安装：
  - 选 `macos-*.dmg`

- 你在 Linux 服务器环境部署：
  - 选 `linux-*.deb / *.rpm / *.tar.gz`

## 推荐使用方式

- 首次部署：
  - 优先使用 `payload` 或平台安装包

- 已有环境升级：
  - 先阅读升级说明，再替换运行包或执行升级流程

- 自定义部署：
  - 使用 `jar + webapps + config` 或完整 `payload`

## 首次安装建议路径

- 想尽量省事：
  - 优先使用安装包或完整 `payload`

- 想保留最大灵活性：
  - 使用 `payload`

- 已经有成熟运维体系：
  - 使用 `jar + 自己的配置与部署脚本`

## 升级建议路径

- 小版本更新：
  - 优先替换对应运行包，并保留旧版本备份

- 涉及安装流程、交付结构或依赖变化的版本：
  - 优先使用完整 `payload` 或新的安装包

- 客户现场升级：
  - 优先使用已验证过的同平台交付制品，避免手工拼装升级

## 已知限制 / 注意事项

- 本项目不是“零依赖即开即用”的单文件工具，实际运行仍依赖 Java、数据库、Redis 或对应的安装准备流程。
- 如果你选择内置 MySQL / Redis 模式，请先确认目标环境允许 Docker，并确保 Docker 本身可安装、可启动、可拉取镜像。
- 不同平台下推荐使用对应平台制品，不建议随意跨平台混用安装包。
- `jar` 适合已有成熟运行环境的用户；如果你需要完整首装能力，优先使用 `payload` 或平台安装包。
- 如需使用安装向导中的环境准备能力，请确认交付包中包含 `scripts/` 与 `installer/` 目录。
- 如果环境中已有现成的 MySQL、Redis、服务注册机制或部署规范，请优先走外部依赖模式，避免重复安装内置依赖。
- 升级前请务必备份现有 `app/`、`config/`、`webapps/` 以及数据库数据。
- 如果版本包含数据库结构变化、依赖模式变化或安装流程变化，不建议直接只替换 Jar，优先按完整交付包升级。
- 授权采用按时长方式，部署前请确认授权周期是否覆盖你的实际使用计划。
- 生产环境发布时，建议同时提供 SHA-256 校验值、变更说明和回滚说明，避免客户现场因版本不一致造成问题。

- 关键附件校验值

```text

payload.zip              SHA256=...

```

## 升级注意事项

- 如涉及数据库结构调整，请先备份数据库
- 如涉及 Redis 配置变更，请确认密码、端口和数据库编号
- 如使用内置依赖模式，请确认 Docker 环境正常
- 升级前建议保留旧版本包，便于回滚

## 回滚建议

如升级失败，建议：

1. 停止当前服务
2. 恢复上一个稳定版本的 `app/`、`config/`、`webapps/`
3. 恢复上一个稳定版本依赖配置
4. 验证健康检查与登录页
